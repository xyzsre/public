Ansible and Terraform for Cisco Network Automation

Understanding the roles, strengths, and integration paths of both tools across IOS, campus LAN, data center, and Meraki environments.

What Is Terraform?

Terraform is an open-source Infrastructure as Code (IaC) platform from HashiCorp. It enables engineers to declare infrastructure resources—networking, compute, storage, security policies—in HashiCorp Configuration Language (HCL) files. Terraform compiles these files into execution plans that provision, modify, and destroy resources in a controlled manner.

Terraform Core Concepts
  • Providers map Terraform resources to APIs (e.g., Cisco, AWS, Azure).
  • State files track deployed resource attributes for drift detection and change planning.
  • Execution plans preview changes before applying them to production infrastructure.

Terraform Use Cases

Multi-Cloud & Hybrid Networking

Provision VPCs/VNets, gateways, and connectivity policies across public clouds and connect them to Cisco SD-WAN or on-premises fabrics.

Infrastructure Lifecycle Automation

Deploy, update, and retire entire network stacks (e.g., Cisco Nexus with ACI/APIC integrations) using version-controlled HCL templates.

Network as Part of DevOps Pipelines

Trigger provisioning from CI/CD workflows alongside application deployments, ensuring consistent network dependencies per environment.

Terraform for Cisco IOS and Switch/Router Configurations

Terraform interacts with Cisco devices through provider plugins. Official and community providers exist for:

  • Cisco IOS XE: Providers leverage RESTCONF/NETCONF APIs to manage interface, routing, and security policy resources.
  • Cisco Meraki: The `cisco/meraki` provider configures networks, SSIDs, firewall rules, and dashboard policies via REST APIs.
  • Cisco Nexus (NX-OS / ACI): Terraform can automate ACI tenants, VRFs, contracts, and fabric policies through the `CiscoDevNet/aci` provider.

Direct Terraform management of classic IOS/IOS XE CLI-based switch and router configurations is limited. Providers focus primarily on API-driven platforms. CLI configuration coverage is more comprehensive in Ansible modules.

Advantages of Ansible for Cisco Automation

Agentless CLI Access

Ansible connects via SSH using modules like cisco.ios.ios_config or cisco.ios.ios_command, ideal for legacy campus switches and routers without REST APIs.

Granular Configuration Management

Role-based playbooks push interface configs, VLANs, ACLs, routing policies, and verification checks, ensuring idempotency and drift detection.

Operational Validation

Playbooks capture show commands, parse outputs, and enforce compliance policies. This is critical for day-2 operations in campus LANs and Nexus data centers.

Why Choose Ansible Instead of Terraform?

  • CLI Coverage: Ansible modules support IOS/IOS XE CLI automation without requiring additional APIs or controllers.
  • Task-Oriented Workflows: Playbooks are ideal for configuration drift remediation, troubleshooting, and incremental change deployment.
  • Immediate Feedback: Gathering outputs and registering results is built in, enabling quick verification after changes.
  • Lower Setup Barrier: Agentless SSH with simple inventory files accelerates pilot projects compared to Terraform state management.

Problem Spaces Solved by Ansible vs Terraform

Ansible Strengths

  • Campus LAN Switch Automation: VLANs, port security, EtherChannel, ACL deployment on Catalyst switches.
  • Routers & WAN Edge: Routing protocol tuning, VRRP/HSRP, QoS policies, and verification commands.
  • Data Center Nexus Operations: NX-OS configuration tasks, VPC management, and show command collection.
  • Meraki Operations: While Terraform handles initial provisioning, Ansible excels at ongoing policy adjustments via APIs.

Terraform Strengths

  • Network Infrastructure Provisioning: Create Meraki networks, ACI tenants, SD-WAN policies, and cloud networking constructs.
  • Infrastructure as Code Governance: Enforce code review and pipeline-driven deployments with immutable plans and state.
  • Cross-Domain Orchestration: Model hybrid architectures linking Cisco fabrics to cloud or security services.
  • Lifecycle Management: Easily destroy or re-create entire environments for testing or DR scenarios.

Combining Ansible and Terraform

A dual-tool strategy often delivers the most value:

  • Use Terraform to provision infrastructure primitives (ACI tenants, Meraki networks, cloud VNets) with versioned state.
  • Employ Ansible playbooks for device-specific CLI configuration, verification, and ongoing compliance checks.
  • Integrate both tools within CI/CD pipelines to provision, configure, and validate Cisco environments end-to-end.

Next Steps

  • Evaluate Providers: Review Terraform provider capabilities (`CiscoDevNet/aci`, `cisco/meraki`, community IOS providers) against your platform requirements.
  • Align Tooling: Decide where Ansible or Terraform best fits within your campus LAN, data center Nexus, and Meraki operations workflows.
  • Adopt GitOps Practices: Store playbooks and Terraform modules in Git, enforce code reviews, and automate testing in lab environments.
  • Iterate in Labs: Use VIRL/GNS3 to validate automation patterns before rolling out to production networks.
